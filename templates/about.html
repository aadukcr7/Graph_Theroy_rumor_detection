About the Project

Why propagation structure matters

This project studies how rumors and organic information spread through temporal propagation trees. It uses interpretable graph-theory and time-aware mathematical features instead of a graph neural network, so every prediction can be explained directly using equations and structural properties of the graph.

Method

Graph generation

Rumor graphs are generated as star-like, hub-heavy, and rapidly branching trees. Organic graphs are generated as longer paths and sparse chains. Both are modeled as temporal directed trees where each edge represents one repost action.

Temporal modeling

Each edge stores delay, timestamp, and temporal weight. Fast reposts receive larger weights, while slow reposts receive smaller weights. This encodes propagation speed directly into the graph structure used for learning.

Classification

A RandomForestClassifier learns nonlinear decision boundaries from the extracted feature vector and predicts whether a cascade is rumor-like or organic, with confidence returned from class probabilities.

Temporal weighting feature

Weight equation

w(i,j) = 1 / ln(1 + delta_t)

Here, delta_t is the repost delay on edge (i,j). If delta_t is small (fast repost), the denominator is small and the weight is large. If delta_t is large (slow repost), the denominator grows and the weight decreases.

Why logarithmic scaling

A logarithm compresses extreme delays so a very late repost does not dominate the entire feature space. This keeps the model numerically stable and preserves meaningful differences among early and medium delays.

How it is used

The system computes average temporal weight across all edges. Rumor-like cascades usually have many early reposts, so their average temporal weight tends to be higher than in slower organic cascades.

Edge-level implementation

  • For each directed edge (u,v), the generator samples a repost delay delta_t.
  • It stores timestamp(v) = timestamp(u) + delta_t.
  • It computes edge weight(u,v) = 1 / ln(1 + delta_t).
  • This weight is saved on that exact edge and sent to the frontend payload.
  • In the graph view, edge thickness is proportional to this weight.
  • On hover, tooltip now shows delay, stored weight, and formula evaluation for that edge.

Worked example on one edge

If one repost has delta_t = 2.00, then weight = 1 / ln(1 + 2.00) = 1 / ln(3.00) approx 0.910. If another edge has delta_t = 6.00, then weight = 1 / ln(7.00) approx 0.514. So the faster edge receives a larger temporal influence in both visualization and model features.

Code mapping

  • Weight computation is implemented in compute_edge_weight(delay).
  • Per-edge attributes delay, timestamp, and weight are attached when edges are created.
  • Average temporal weight and diffusion speed are derived from edge delays and weights in feature extraction.

Mathematics used in this project

Graph structure metrics

  • Density: 2|E| / (|V|(|V|-1)) for undirected view of the cascade.
  • Diameter: max shortest-path distance between node pairs.
  • Radius: minimum eccentricity across nodes.
  • Center: nodes whose eccentricity equals the radius.
  • Average shortest path: mean of shortest-path lengths over node pairs.

Centrality and shape

  • Degree centrality: deg(v) / (|V|-1).
  • Betweenness centrality: fraction of shortest paths that pass through a node.
  • Closeness centrality: inverse of average distance from a node to others.
  • Degree centralization: sum(max(CD) - CD(v)) / (|V|-2), a Freeman-style score.
  • Branching factor and leaf ratio quantify tree spread shape and terminal-node concentration.

Time-to-diffusion mathematics

  • Each edge stores delay delta_t and cumulative timestamp t(parent)+delta_t.
  • Average temporal weight summarizes urgency of repost behavior.
  • Diffusion speed is approximated as 1 / mean(delta_t), with a tiny epsilon for stability.
  • This combines topological and temporal signals in one feature vector.
